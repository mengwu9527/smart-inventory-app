## 完整更新流程

### 步骤1：确认版本号已更新 ✅

已将 `version` 从 `1.0.0` 改为 `1.1.0`。

### 步骤2：构建新的APK

**在Windows 10/11环境中执行**：

```cmd
# 1. 进入项目目录
cd E:\path\to\project\client

# 2. 连接项目到Expo（如果未连接）
npx eas-cli@latest init --id f05dfeb3-bc5b-42c0-b267-5084f48f7014

# 3. 构建APK
npx eas-cli@latest build --platform android --profile preview
```

**预计时间**：10-15分钟

### 步骤3：下载新APK

构建完成后：

1. 从EAS构建页面下载APK
2. 或从控制台输出的链接下载

### 步骤4：安装更新

**方法1：直接安装（推荐）**

```
1. 将APK传输到Android设备
2. 在设备上打开APK
3. 点击安装
4. 系统自动：
   - 卸载旧版本
   - 保留应用数据（数据库、缓存、设置）
   - 安装新版本
```

**方法2：使用ADB安装**

```cmd
# 连接Android设备
adb devices

# 安装APK（自动更新）
adb install -r app-debug.apk
```

`-r` 参数表示替换已安装的应用（保留数据）。

### 步骤5：验证更新

在设备上检查：

1. **打开应用**
   - 应用名称：智慧记AI进销存
   - 版本号：1.1.0

2. **验证功能**
   - 检查新功能是否正常
   - 检查数据是否保留
   - 检查设置是否保留

## 更新 vs 重新安装

### 更新（推荐）✅

**前提条件**：
- ✅ 相同的包名（已满足）
- ✅ 更高的versionCode（已满足）
- ⚠️ 相同的签名密钥（需确认）

**优点**：
- ✅ 数据保留（数据库、缓存、用户数据）
- ✅ 设置保留
- ✅ 无需重新配置
- ✅ 用户体验更好

**缺点**：
- ⚠️ 需要满足签名要求

### 重新安装（备选）❌

**操作**：
```cmd
# 先卸载旧版本
adb uninstall com.free.jxc.app

# 再安装新版本
adb install app-debug.apk
```

**优点**：
- ✅ 无需满足签名要求
- ✅ 无需满足版本号要求

**缺点**：
- ❌ 数据全部丢失
- ❌ 设置需要重新配置
- ❌ 用户需要重新登录
- ❌ 体验差

## 签名密钥问题

### 签名密钥的重要性

Android应用必须用密钥签名，系统根据签名判断：
- 是否允许更新（必须相同签名）
- 应用是否可信
- 应用是否来自同一开发者

### EAS构建的签名处理

#### 情况1：之前用EAS构建 ✅

**判断标准**：
- 旧APK的构建记录在EAS平台
- 使用相同的项目ID（f05dfeb3-bc5b-42c0-b267-5084f48f7014）

**结果**：
- ✅ EAS会自动使用相同的密钥
- ✅ 可以正常更新
- ✅ 无需额外操作

#### 情况2：之前手动构建 ⚠️

**判断标准**：
- 旧APK是用本地Keystore构建的
- 项目中有keystore文件
- 有构建脚本

**结果**：
- ⚠️ 需要将Keystore上传到EAS
- ⚠️ 或配置EAS使用该Keystore
- ⚠️ 否则无法更新（只能重新安装）

#### 情况3：未知构建方式 ⚠️

**如果不确定旧APK是如何构建的**：

**方法1：测试构建**
1. 使用EAS构建新APK
2. 尝试安装更新
3. 如果成功，说明密钥匹配
4. 如果提示"签名不匹配"，需要处理

**方法2：查看APK信息**
```cmd
# 使用aapt查看APK信息
aapt dump badging app.apk

# 查看签名信息
keytool -printcert -jarfile app.apk
```

### 如何确认旧APK的构建方式

#### 检查项目文件

```cmd
# 检查是否有keystore文件
dir *.jks
dir *.keystore

# 检查构建脚本
dir build.gradle
dir gradle.properties

# 检查EAS配置
type eas.json
```

#### 查看EAS构建历史

1. 访问 https://expo.dev
2. 登录Expo账号
3. 进入项目页面
4. 查看构建历史
5. 确认之前的构建是否使用EAS

### 签名不匹配的解决方案

#### 方案A：上传Keystore到EAS（推荐）

**如果旧APK是用本地Keystore构建的**：

1. **获取Keystore文件**
   - 从构建环境获取
   - 或从备份恢复

2. **上传到EAS**
   ```cmd
   npx eas-cli@latest credentials
   ```

3. **选择"Upload a new keystore"**
   - 按提示上传
   - 输入密码

4. **重新构建APK**
   ```cmd
   npx eas-cli@latest build --platform android --profile preview
   ```

#### 方案B：重新安装（备选）

如果无法获取旧Keystore：

```cmd
# 卸载旧版本
adb uninstall com.free.jxc.app

# 安装新版本
adb install app-debug.apk
```

**缺点**：数据丢失

## 常见问题

### Q1：安装时提示"签名不匹配"

**原因**：
- 新旧APK使用了不同的签名密钥

**解决方案**：
1. 确认旧APK的构建方式
2. 如果是EAS构建，使用相同的项目ID
3. 如果是手动构建，上传Keystore到EAS
4. 或选择重新安装（数据会丢失）

### Q2：安装时提示"版本号过低"

**原因**：
- 新APK的versionCode不大于旧APK

**解决方案**：
- 增加 `version`（如 1.1.0 → 1.1.1）
- 重新构建APK

### Q3：安装时提示"包名不匹配"

**原因**：
- 新旧APK的包名不同

**解决方案**：
- 检查 `app.config.ts` 中的 `android.package`
- 确保与旧APK的包名一致

### Q4：更新后数据丢失

**原因**：
- 签名不匹配（导致重新安装）
- 或数据库结构变化

**解决方案**：
- 确保使用相同的签名
- 实现数据库迁移逻辑

### Q5：无法卸载旧版本

**原因**：
- 应用是系统应用
- 或设备权限限制

**解决方案**：
- 确保设备已Root（系统应用需要）
- 或使用ADB卸载：`adb uninstall com.free.jxc.app`
